# 模式切换逻辑优化 - 支持跨模式对话

## 问题描述

**用户反馈**：
- ❌ 在"询问"模式对话后，无法切换到"处理"模式进行修改
- ❌ 模式锁定过于严格，限制了使用灵活性
- ✅ 希望：对话完成后可以切换模式，消息在两种模式间共享

## 解决方案

### 1. 放宽模式切换限制

**之前的逻辑**：
```kotlin
// ❌ 只要有消息就锁定模式
fun setMode(quickMode: QuickAiMode) {
    if (_conversationHistory.value.isEmpty()) {
        _currentMode.value = quickMode
    }
}
```

**现在的逻辑**：
```kotlin
// ✅ 只在加载中才锁定，完成后可以切换
fun setMode(quickMode: QuickAiMode) {
    if (!_isLoading.value) {
        _currentMode.value = quickMode
    }
}
```

### 2. UI 反馈优化

**按钮状态**：
- ⚡ **加载中**：按钮禁用（灰色）- 防止切换导致混乱
- ✅ **完成后**：按钮可用 - 允许切换模式继续对话

```kotlin
FilterChip(
    selected = currentMode == QuickAiMode.AI_QUERY,
    onClick = { viewModel.setMode(QuickAiMode.AI_QUERY) },
    enabled = !isLoading  // 只在加载时禁用
)
```

### 3. 始终显示选中文本

**优化**：
- 选中的文本始终显示在对话历史上方
- 方便在切换模式后参考原文
- 有对话历史时添加分隔线

```kotlin
// 始终显示选中文本（不管是否有消息）
item {
    Text(text = if (currentMode == QuickAiMode.AI_QUERY) "关于：" else "选中的文本：")
    Surface { Text(text = editableSelectedText) }
    
    if (hasMessages) {
        HorizontalDivider()  // 有历史时添加分隔线
    }
}
```

## 注意事项

### 为什么加载中要锁定？

**原因**：
- AI 正在基于当前模式生成回复
- 切换模式会导致系统提示词不匹配
- 可能产生错误或不符合预期的回复

**示例**：
```
❌ 错误场景：
1. [询问模式] 发送："这段话有什么问题？"
2. AI 正在思考...
3. 切换到 [处理模式]  // ← 不应该允许
4. AI 回复：修改后的文本  // ← 但用户期待的是问题分析
```

### 最佳实践

1. **等待回复完成**：让 AI 完成当前回复后再切换模式
2. **明确指令**：切换后明确告诉 AI 你的新需求
3. **参考历史**：利用对话历史作为上下文
